# LSS ERP DB Schema Tuning Review

작성일: 2026-06-08

## 결론

운영 전 DB 스키마 튜닝은 필요하다. 현재 스키마는 ERP 전체 메뉴를 빠르게 구현하기 위해 넓게 잡힌 상태이며, 실제 확정 업무 흐름보다 테이블과 컬럼이 먼저 만들어진 부분이 있다. 다만 대부분의 테이블은 이미 라우터와 화면에 연결되어 있으므로, 즉시 삭제보다는 `기초 -> 영업 -> 실행 -> 구매/회계 -> 경영` 순서로 업무 범위를 확정한 뒤 Alembic migration으로 단계적으로 줄이는 것이 맞다.

## 우선순위

### P0: 운영 전 반드시 정리

| 대상 | 이슈 | 권장 조치 |
| --- | --- | --- |
| `execution.py`, `management.py`, `timesheet.py`, `vehicle.py` 일부 기본값/주석 | 한글 인코딩이 깨진 문자열이 모델 기본값으로 들어가 있다. 데이터 저장 시 깨진 상태값이 실제 DB에 들어갈 수 있다. | 모델 기본값과 주석을 정상 한글/영문 코드값으로 정리하고 migration 생성 |
| `projects` vs `sites`/`contracts` | 실행 프로젝트와 현장/계약 테이블이 업무적으로 겹친다. 화면은 `projects`도 쓰고, 영업/구매는 `sites`와 `contracts`도 쓴다. | “프로젝트”와 “현장”의 기준 엔티티를 하나로 정하거나 명확히 역할 분리 |
| `sales_bills`/`ap_bills` vs `progress_billings`/`accounts_receivable`/`accounts_payable` | 매출/매입 청구 정보가 여러 테이블에 분산되어 중복될 가능성이 크다. | 청구 원장과 회계 채권/채무 생성 흐름을 확정 |
| `companies` 86개 컬럼 | 거래처 등록 화면 중심으로 많은 코드/명칭 쌍이 한 테이블에 들어가 있다. 향후 담당자, 수신처, 계좌가 여러 건이면 확장 어렵다. | 운영 요구가 1건 고정이면 유지, 다건이면 `company_contacts`, `company_receivers`, `company_bank_accounts` 분리 |

### P1: 업무 확정 후 축소 후보

| 대상 | 이슈 | 권장 조치 |
| --- | --- | --- |
| `purchase_contracts` vs `subcontracts` | 실행 구매계약과 외주계약 테이블이 일부 역할이 겹친다. | 구매계약/외주계약을 하나의 계약 테이블로 통합할지 결정 |
| `cost_inputs` | 원가 통합 입력 테이블이 있으나 노무/경비/입고/외주 등 원천 테이블과 중복 집계가 될 수 있다. | 집계 캐시인지, 수기 원가 입력인지 역할 명확화 |
| `inventory.current_qty`, `reserved_qty`, `available_qty` | 재고 수량은 `inventory_transactions`에서 계산 가능하다. 저장 컬럼이면 정합성 관리가 필요하다. | 운영 성능상 캐시가 필요하면 유지하고 트랜잭션 생성 시 갱신 규칙 고정 |
| `revenue_forecasts`, `fund_plans`, `dept_budgets` | 계획/예측 테이블들은 기능은 분리되어 있으나 공통 구조가 많다. | 현 단계에서는 유지, 운영 후 중복 패턴이 확정되면 통합 검토 |
| `unit_prices` | 현재는 기준 단가만 단순 관리한다. 거래처별/기간별/프로젝트별 단가 정책이 확정되지 않았다. | 단가 정책 확정 전까지 최소 필드 유지 |

### P2: 유지 가능하지만 제약 보강 필요

| 대상 | 이슈 | 권장 조치 |
| --- | --- | --- |
| 코드성 컬럼 `status`, `type`, `*_yn` | 대부분 자유 문자열이다. 잘못된 값 저장 가능성이 있다. | PostgreSQL `CHECK` 또는 코드 테이블 도입 검토 |
| `ref_type`, `ref_id` 패턴 | FK가 없어 참조 무결성을 DB가 보장하지 못한다. | 정말 다형 참조가 필요한 곳만 사용하고, 핵심 흐름은 명시 FK로 변경 |
| 금액 컬럼 | 계산 컬럼과 입력 컬럼이 섞여 있다. | 입력값/계산값/캐시값 구분 및 갱신 책임 정의 |
| `created_by`, `approved_by` 등 사용자 참조 | 일부는 FK이나 관계/인덱스가 일관되지 않다. | 조회가 많은 FK에 인덱스 보강 |

## 삭제를 바로 하면 안 되는 이유

- 프론트 라우터에 대부분의 기능 화면이 이미 연결되어 있다.
- 백엔드 라우터도 각 테이블에 대한 CRUD 또는 집계 API를 제공한다.
- 현재 사용 여부만 보고 삭제하면 메뉴 진입, 대시보드, 경영 분석 API가 함께 깨질 수 있다.
- Alembic baseline이 생성된 직후라, 삭제/통합은 별도 migration으로 의도를 남기는 편이 추적 가능하다.

## 권장 진행 순서

1. 현재 개발 우선순위인 `기초 > 거래처 관리`, `기초 > 자재 관리`의 운영 필드를 먼저 확정한다.
2. `companies`, `materials`에서 화면에 없거나 정책 미확정인 컬럼을 `보류 컬럼`으로 표시한다.
3. 프로젝트/현장/계약의 기준 엔티티를 확정한다.
4. 매출/매입 청구와 회계 채권/채무 생성 흐름을 확정한다.
5. 확정된 변경만 Alembic migration으로 반영한다.
6. migration 후 `docs/LSS_ERP_DB_테이블_명세서.xlsx`를 재생성한다.

## 즉시 적용해도 안전한 작업

- 깨진 한글 기본값/주석 정리
- 상태값을 한글 표시값이 아닌 영문 코드값으로 통일
- `AUTO_CREATE_SCHEMA=false` 운영 기준 유지
- 신규 컬럼 추가 전 Alembic migration 생성 규칙 고정
- DB 명세서에 “사용 여부/확정 여부/비고” 컬럼 추가

## 보류해야 하는 작업

- 테이블 삭제
- `projects`, `sites`, `contracts` 통합
- 청구/채권/채무 테이블 통합
- 거래처 담당자/계좌/수신처 테이블 분리

위 작업은 실제 업무 프로세스가 확정된 후 진행해야 한다.
